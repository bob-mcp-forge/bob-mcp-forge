Overview

Bob MCP Forge is an AI-powered developer tool that converts plain English descriptions into production-ready Model Context Protocol (MCP) server code. Simply describe what you want your MCP tool to do, and Bob MCP Forge generates complete TypeScript code with built-in compliance checks and one-click deployment to IBM Cloud.

What is MCP?
Model Context Protocol (MCP) is an open protocol that enables AI assistants like Claude to securely connect to external data sources and tools. MCP servers provide tools, resources, and prompts that AI models can use.

The Enterprise Trust Gap

Bob MCP Forge addresses a critical enterprise challenge: organizations face two major obstacles when adopting MCP technology.

  • Non-developers cannot easily build MCP tools: Creating MCP servers requires deep technical knowledge of TypeScript, the MCP protocol, and best practices. Business users and domain experts who understand the requirements cannot translate their needs into working code.
  • Security officers distrust unverified AI-generated code: While AI can generate code quickly, enterprises cannot deploy it without rigorous security scanning, compliance validation, and governance oversight. The risk of vulnerabilities, data leaks, or policy violations is too high.

How Bob MCP Forge Solves This

Bob MCP Forge bridges the Trust Gap by combining AI-powered creation with enterprise-grade governance in a single platform:

1. Plain English Prompt

Anyone can describe what they need in natural language. No coding required.

2. Bob Generation

AI generates production-ready TypeScript code following MCP protocol standards and security best practices.

3. Compliance Scan

Automated security, performance, and policy checks identify issues before deployment. Supports GDPR, SOC 2, and HIPAA profiles.

4. Apply Fix Remediation

One-click fixes for compliance issues. No manual code editing required.

5. One-Click IBM Cloud Deployment

Containerized deployment to IBM Cloud with automatic provisioning, health checks, and monitoring.

This end-to-end workflow ensures that MCP servers are both easy to create and safe to deploy, making AI-powered development accessible to enterprises without compromising security or compliance.

Getting Started

Follow these steps to create your first MCP server with Bob MCP Forge:

  1. Describe Your Tool
    Navigate to the landing page and enter a description of what you want your MCP server to do. Be specific about the functionality, data sources, and any special requirements.
  2. Select Compliance Profile
    Choose a compliance profile that matches your requirements: General, GDPR, SOC 2, or HIPAA. This ensures your generated code meets specific regulatory standards.
  3. Generate Code
    Click "Generate MCP Server" and wait 15-30 seconds while the AI generates your complete server code, including TypeScript implementation, schema definitions, and Docker configuration.
  4. Review Code
    Browse the generated files in the code preview screen. You can view server.ts, schema.json, package.json, Dockerfile, and README.md.
  5. Check Compliance
    Review the compliance dashboard to see security, performance, and policy issues. Apply suggested fixes for any critical or warning items.
  6. Deploy
    Configure your deployment settings (region, instance size) and click "Deploy to IBM Cloud" to make your MCP server live.
Quick Start

System Architecture

Bob MCP Forge is built as a monorepo with clearly separated concerns, enabling scalability, maintainability, and independent deployment of components.

Architecture Components

Frontend UI (React)

User interface for describing tools, viewing generated code, reviewing compliance reports, and managing deployments. Built with React and modern CSS.

Orchestrator API (Express)

RESTful API that coordinates generation requests, manages job lifecycle, and handles deployment operations. Exposes endpoints for frontend communication.

Generator Service (TypeScript)

Core generation engine that transforms natural language descriptions into MCP server code. Uses string templates and Zod validation for type-safe generation.

Job Store (SQLite / Memory)

Persistent storage for generation jobs, status tracking, and metadata. Supports both SQLite for production and in-memory for development.

Generated MCP Server Artifacts

Output files including server.ts, schema.json, package.json, Dockerfile, and README.md. Ready for deployment or local development.

Request Flow

When a user generates an MCP server, the following sequence occurs:

  1. Frontend sends POST /api/v1/generate
    User submits tool description and compliance profile to the Orchestrator API.
  2. Orchestrator triggers generation
    API validates the request, creates a job record, and invokes the Generator Service.
  3. Generator emits MCP server files
    Generator Service processes the description, applies templates, and produces all necessary files.
  4. Job persisted to store
    Generated files and metadata are saved to the Job Store for retrieval and deployment.

Generator Internals

The Generator Service uses a template-based approach with validation:

  • String Templates: Pre-built templates for common MCP patterns (API integration, database access, file operations)
  • Zod Validation: Runtime type checking ensures generated code matches expected schemas
  • Security Policies: Automatic injection of input validation, error handling, and rate limiting
  • Best Practices: TypeScript strict mode, structured logging, and comprehensive error messages

Bob MCP Forge Monorepo Architecture

Generation Pipeline

The Generator Service follows a multi-stage pipeline to transform natural language descriptions into production-ready MCP servers.

Pipeline Stages

1 Input Manager

Receives and sanitizes the tool description. Extracts key requirements such as tool names, parameters, data sources, and authentication needs. Validates input against security policies.

2 Template Library

Provides base templates for different MCP server types. Includes templates for API integrations, database queries, file operations, and custom business logic. Each template follows MCP protocol standards.

3 Template Engine

Applies security policies and best practices to templates. Injects input validation, error handling, rate limiting, and logging. Ensures compliance with selected profile (GDPR, SOC 2, HIPAA).

4 Output Formatter

Emits final files with proper formatting and structure. Generates server.ts, schema.json, package.json, Dockerfile, and README.md. Ensures all files are syntactically correct and ready for deployment.

MCP Generator Internal Pipeline

End-to-End Flow

From developer input to running MCP server:

  1. Developer/LLM provides tool description
    Natural language description of desired MCP server functionality, including tools, parameters, and requirements.
  2. MCP Generator applies templates and logic
    Generator Service processes the description through the pipeline stages, applying templates, security policies, and best practices.
  3. Generated project files are emitted
    Complete MCP server project with all necessary files (TypeScript source, configuration, Docker setup, documentation).
  4. Build System runs npm install && npm run build
    Dependencies are installed and TypeScript is compiled to JavaScript. Build artifacts are prepared for execution.
  5. MCP Host registers server over stdio
    MCP host (like Claude Desktop) connects to the server via standard input/output, establishing the communication channel.
  6. Tool can be invoked via MCP
    AI assistant can now discover and invoke the tools provided by the MCP server, enabling new capabilities.
Pipeline Performance
The entire generation pipeline typically completes in 15-30 seconds, depending on the complexity of the tool description and the number of tools being generated.

Compliance

Bob MCP Forge performs comprehensive compliance scanning on all generated code. Issues are categorized by severity:

CRITICAL Critical Issues

Security vulnerabilities or compliance violations that must be fixed before deployment. Examples: missing input validation, exposed API keys, authentication bypasses.

WARNING Warning Issues

Best practice violations or potential problems that should be addressed. Examples: missing error handling, no rate limiting, inadequate logging.

INFO Info Issues

Suggestions for improvement or optional enhancements. Examples: TypeScript strict mode, structured logging, additional documentation.

Common Compliance Issues

  • Missing Input Validation: User inputs not validated against schema
  • API Key Exposure: Credentials hardcoded in source code
  • Missing Error Handling: API calls not wrapped in try-catch blocks
  • No Rate Limiting: Unlimited API requests allowed
  • Inadequate Logging: No structured logging for debugging

Applying Fixes

For each issue, Bob MCP Forge provides:

  • Detailed explanation of the problem
  • File location and line numbers
  • Suggested fix with code examples
  • "Apply Fix" button for one-click resolution

Generated File Structure

Every MCP server generated by Bob MCP Forge includes these files:

server.ts

Main TypeScript implementation of your MCP server. Includes:

import { Server } from '@modelcontextprotocol/sdk/server/index.js'; import { StdioServerTransport } from '@modelcontextprotocol/sdk/server/stdio.js'; // Server initialization const server = new Server({ name: 'your-mcp-server', version: '1.0.0' }, { capabilities: { tools: {} } }); // Tool definitions and handlers server.setRequestHandler(ListToolsRequestSchema, async () => ({ tools: [/* your tools */] }));

schema.json

MCP protocol schema defining your tools, resources, and prompts:

{ "name": "your-mcp-server", "version": "1.0.0", "tools": [ { "name": "tool_name", "description": "Tool description", "inputSchema": { /* JSON Schema */ } } ] }

package.json

Node.js package configuration with dependencies and scripts:

{ "name": "your-mcp-server", "version": "1.0.0", "type": "module", "dependencies": { "@modelcontextprotocol/sdk": "^1.0.0" }, "scripts": { "build": "tsc", "start": "node dist/server.js" } }

Dockerfile

Container configuration for deployment:

FROM node:20-alpine WORKDIR /app COPY package*.json ./ RUN npm install COPY . . RUN npm run build CMD ["npm", "start"]

README.md

Documentation including setup instructions, usage examples, and API reference.

Deployment

Bob MCP Forge makes deployment to IBM Cloud simple and fast. The deployment process includes:

Pre-Deployment Checklist

Before deployment, Bob MCP Forge verifies:

  • ✅ Code successfully generated
  • ✅ All dependencies validated
  • ✅ Dockerfile created
  • ✅ Environment variables configured
  • ✅ No critical compliance issues

Configuration Options

  • Region: Choose from us-south, us-east, eu-gb, or eu-de
  • Instance Size: Small (1 vCPU, 2GB), Medium (2 vCPU, 4GB), or Large (4 vCPU, 8GB)
  • Server Name: Custom name for your deployment

Deployment Process

  1. Building Docker image from your code
  2. Pushing image to IBM Cloud Registry
  3. Creating Cloud Foundry application
  4. Deploying application to selected region
  5. Running health checks
  6. Providing live URL

Typical deployment time: 2-3 minutes

Post-Deployment

After successful deployment, you'll receive:

  • Live URL for your MCP server
  • Status indicators (running, region, instance)
  • Access to deployment logs
  • Options to create new servers or view dashboard

Troubleshooting

Generation Issues

Problem: Generation takes too long or times out
Solution: Simplify your prompt or break it into smaller components. Try being more specific about the core functionality.
Problem: Generated code doesn't match requirements
Solution: Provide more detailed specifications in your prompt. Include specific API endpoints, data formats, and error handling requirements.

Compliance Issues

Problem: Too many critical compliance issues
Solution: Use the "Apply Fix" buttons to automatically resolve issues. Review the suggested fixes and apply them one by one.
Problem: Can't deploy due to compliance failures
Solution: All critical issues must be resolved before deployment. Focus on security issues first (input validation, API key exposure).

Deployment Issues

Problem: Deployment fails during build
Solution: Check the deployment logs for specific errors. Common issues include missing dependencies or invalid Dockerfile configuration.
Problem: Server not responding after deployment
Solution: Verify environment variables are set correctly. Check health check logs in the deployment dashboard.
